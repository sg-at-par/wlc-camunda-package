④ Configurable Data Persistence Router
🗄 Platform Storage
  • mode = platform (default)
  • Writes domain object fields as JSONB → step_submissions.platform_data
  • Keyed by domain object ID within the JSONB column
  • Atomic per domain object — all fields or none
  • Zero subscriber infrastructure required
💾 External Storage
  • mode = external_db
  • Writes directly to subscriber-configured database
  • Supported: PostgreSQL · MySQL · MSSQL · MongoDB · Oracle
  • Connection string AES-256 encrypted in domain_object_configs
  • Field-ID → column/field mapping spec per domain object
  • Per-subscriber isolated connection pool — no cross-subscriber reuse
  • Data written ONLY to external DB — never copied to platform_data
  • routing_outcomes records receipt only (destination + status + timestamp)
  • Failure → synchronous error → user re-submits (no DLQ in Phase 1)
  • External API mode (REST/webhooks) → Phase 2
⚡ Data Sovereignty Rule (DC-04):  Domain objects with mode=external_db are written EXCLUSIVELY to the subscriber's external database. The platform never holds a copy. step_submissions.platform_data is only populated for mode=platform domain objects. routing_outcomes records destination + status + timestamp — never field values.
⛔ Phase 1 Failure Behaviour:  Persistence is synchronous. If any domain object write fails (platform or external DB), the entire submission is rejected. Camunda task is NOT completed. User sees the error and must re-submit. No DLQ or async retry in Phase 1.
All persistence succeeded synchronously → POST Camunda task complete → write audit log → poll next task

⚠ Design Considerations
DC-01 — form_definition naming
Column renamed from schema → form_definition
schema is overloaded — it means something specific in PostgreSQL namespacing, JSON Schema validation, and database design. form_definition unambiguously identifies this as the JSON descriptor for the step's UI rendering.
DC-02 — domain_objects placement
domain_objects in workflow_steps is a coupling concern
domain_objects defines which fields belong to which persistence group — a persistence concern, not a form rendering concern. Currently stored in workflow_steps for Phase 1 simplicity. Recommended Phase 2 refactor: move to a dedicated step_domain_object_mappings table linking workflow_steps to domain_object_configs.
DC-03 — available_actions source
available_actions ideally derived from Camunda BPMN
Ideally available_actions (ESCALATE, CLOSE, SEND_BACK) should come from Camunda's outgoing sequence flows on a user task — the BPMN owns that truth. However, Camunda 8.7 Task API does not expose available gateway paths directly. Storing in workflow_steps is a known workaround. Revisit if Camunda exposes this in a future version.
DC-04 — step_submissions data sovereignty
step_submissions is NOT a universal data store
platform_data stores ONLY fields of domain objects with mode=platform. Domain objects with mode=external_db are written EXCLUSIVELY to the external database — never duplicated here. routing_outcomes records only the routing receipt (destination, status, timestamp) — never field values. The subscriber's external database is the sole system of record for externally routed data.
DC-05 — bpmn_resource_name purpose
bpmn_resource_name is a reference field, not operational
Stores the BPMN filename (e.g. soc-alert-investigation.bpmn) as deployed to Camunda. Audit/reference field only — the BPMN XML lives inside Camunda's internal store, not here. Phase 1: populated manually when seeding this table. Phase 2: auto-populated by Subscriber Onboarding Service on BPMN upload.
DC-06 — subscribers vs tenants naming
Table named subscribers — not tenants
Deliberately avoids the word "tenant" to prevent confusion with Camunda's own multi-tenancy concept. In this platform, the subscribing organisation is called a subscriber. All FK references use subscriber_id. Camunda's tenant feature is a separate, unrelated concept.

📤step_submissions submission record ▶
Submission record and platform data store. Holds ONLY domain object data routed to platform storage (mode=platform). External DB domain object data is NEVER stored here — only a routing receipt. Also mirrors Camunda orchestration variables for audit purposes.
idUUIDPK
subscriber_idUUIDFK → subscribers.id
instance_idUUIDFK → process_instances.id
step_idUUIDFK → workflow_steps.id
submitted_byTEXTuser identifier
submitted_atTIMESTAMPTZDEFAULT now()
platform_dataJSONBmode=platform domain objs ONLY
routing_outcomesJSONBreceipt only — no field values (DC-04)
camunda_variablesJSONBaudit mirror of camundaVar=true fields
overall_statusTEXTsuccess | partial | failed
⚠ DC-04: platform_data NOT populated for external_db domain objects. routing_outcomes = destination + status + timestamp only — never field values.
CREATE TABLE step_submissions (
  id                UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  subscriber_id     UUID        NOT NULL REFERENCES subscribers(id),
  instance_id       UUID        NOT NULL REFERENCES process_instances(id),
  step_id           UUID        NOT NULL REFERENCES workflow_steps(id),
  submitted_by      TEXT,
  submitted_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  platform_data     JSONB       DEFAULT '{}',
  -- DC-04: ONLY populated for mode=platform domain objects
  -- {"analyst-record": {"analystName":"...", "notes":"..."}}
  routing_outcomes  JSONB       DEFAULT '{}',
  -- DC-04: receipt only — NO field values for external_db objects
  -- {"incident-record": {mode:"external_db",status:"success",at:"..."}}
  camunda_variables JSONB       DEFAULT '{}',
  -- audit mirror of camundaVar=true orchestration fields sent to Camunda
  overall_status    TEXT        NOT NULL DEFAULT 'success'
                    CHECK (overall_status IN ('success','partial','failed'))
);
CREATE INDEX idx_ss_instance
  ON step_submissions (subscriber_id, instance_id);
📜wlc_audit_log append-only · immutable ▶
Immutable, append-only event log for the entire platform. Records every significant event — submissions, routing decisions, persistence outcomes, Camunda completions, access denials, and failures. Cannot be updated or deleted (enforced by DB rules).
idBIGSERIALPK — order-guaranteed serial
subscriber_idUUIDFK → subscribers.id
instance_idUUIDFK → process_instances nullable
event_typeTEXTNOT NULL
routing_decisionJSONBdomain → mode mapping
outcomeJSONBsuccess/fail per domain obj
actorTEXTuser or system
occurred_atTIMESTAMPTZNOT NULL DEFAULT now()
DB RULEs block UPDATE + DELETE — table is append-only by enforcement
CREATE TABLE wlc_audit_log (
  id               BIGSERIAL    PRIMARY KEY,
  subscriber_id    UUID         NOT NULL REFERENCES subscribers(id),
  instance_id      UUID         REFERENCES process_instances(id),
  event_type       TEXT         NOT NULL,
  -- 'STEP_SUBMITTED' | 'PERSISTENCE_ROUTED' | 'CAMUNDA_COMPLETED'
  -- 'PERSIST_FAILED' | 'ACCESS_DENIED' | 'INSTANCE_STARTED'
  routing_decision JSONB,
  outcome          JSONB,
  actor            TEXT,
  occurred_at      TIMESTAMPTZ  NOT NULL DEFAULT now()
);
-- Enforce append-only: block UPDATE and DELETE
CREATE RULE no_update_audit AS ON UPDATE TO wlc_audit_log
  DO INSTEAD NOTHING;
CREATE RULE no_delete_audit AS ON DELETE TO wlc_audit_log
  DO INSTEAD NOTHING;
CREATE INDEX idx_audit_subscriber_time
  ON wlc_audit_log (subscriber_id, occurred_at DESC);
