WLC API Server
Express.js · REST boundary · Port 3001
  • HTTP boundary only — no business logic
  • Routes template requests → Form Template Engine
  • Routes submissions → Submission Orchestrator
  • Routes lifecycle ops → Workflow Lifecycle Manager
  • Enforces CORS · attaches X-Correlation-ID
📋
Form Template Engine
GET /api/workflows/:key/steps/:stepId/template
  • Loads form_definition JSONB from workflow_steps
  • Evaluates showWhen conditional visibility rules
  • Pre-fills fields on SEND_BACK from prior submission
  • Returns availableActions to constrain action buttons
  • Read-only — never writes to any DB
📤
Submission Orchestrator
POST /api/instances/:instanceId/submit
  • Splits payload on camundaVar flag
  • camundaVar=true → orchestration vars → Workflow Lifecycle Manager
  • camundaVar=false → domain object data → WLC PostgreSQL
  • Persistence synchronous — DB write before Camunda call
  • Writes audit log entry in same atomic transaction
🔄
Workflow Lifecycle Manager
Instance start · poll · recover · cancel
  • POST → starts Camunda instance · registers in process_instances
  • After submission: polls GET /v1/user-tasks for next taskDefinitionId
  • taskDefinitionId → triggers Form Template Engine lookup
  • GET /v1/process-instances/:key → state check on recovery
  • DELETE /v1/process-instances/:key → cancel on request
📜
Audit Log Writer
wlc_audit_log — append-only · immutable
  • Every submission → immutable audit entry
  • Records routing decision: orchestration vars + domain object data
  • Records persistence outcome synchronously
  • PostgreSQL rules block UPDATE and DELETE
  • log_audit_event() stored function — only write path
⚡ Submission Rule: Persistence is fully synchronous. If WLC DB write fails, submission is rejected — user must re-submit. Camunda task is NOT completed until all writes succeed.
camundaVar=true → orchestration vars · camundaVar=false → domain object data → WLC PostgreSQL

Workflow Registry
camunda_workflows · workflow_steps
  • camunda_workflows — process definition keys + metadata
  • workflow_steps — form_definition JSONB per step (GIN indexed)
  • Source of truth for Form Template Engine

Domain Object Store
process_instances · step_submissions
  • process_instances — runtime state + current step tracking
  • step_submissions.wlc_data — all camundaVar=false field values (JSONB)
  • step_submissions.camunda_variables — orchestration vars audit mirror
  • Immutable once written — no UPDATE on step_submissions
🔒
Audit Store
wlc_audit_log — UPDATE/DELETE blocked by PG rules
  • 18 event types via audit_event_type enum
  • PostgreSQL rules structurally prevent mutation
  • log_audit_event() stored function — only permitted write path
  • GIN index on payload JSONB for fast incident queries
⛔ Hard Isolation Boundary: WLC PostgreSQL has NO shared tables with Camunda. Camunda engine has ZERO access to this database. Domain object data NEVER crosses into Camunda — ever.
All WLC writes succeeded synchronously → POST Camunda task complete (orchestration vars only) → write audit → poll next task

④ Camunda 8.7 Domain — On-Premises · Hard Isolation Boundary
🔗
WLC ↔ Camunda API Bridge — 5 Calls · Workflow Lifecycle Manager Only
Only orchestration variables cross this boundary · domain object data NEVER crosses
POST/v1/process-instancesStart Process Instance — initiates alert investigation
Body: { processDefinitionKey, variables: { initiatedBy } } · Returns: processInstanceKey → stored in process_instances
GET/v1/user-tasks?processInstanceKey=…Get Next User Task — poll after each submission
Returns: { taskId, taskDefinitionId, candidateGroups } · taskDefinitionId → next form_definition lookup
POST/v1/tasks/:taskId/completeComplete User Task — advance workflow
Body: { variables: { action, severity, … } } · ONLY camundaVar=true orchestration fields · domain object data NEVER sent
GET/v1/process-instances/:keyGet Process Instance State — recovery when WLC state is stale
Returns: { state: ACTIVE | COMPLETED | CANCELLED, variables }
DEL/v1/process-instances/:keyCancel Process Instance — withdraw or duplicate alert
Sets process_instances.status = TERMINATED in WLC DB · writes PROCESS_TERMINATED audit event
⚠ Orchestration vs Domain Object split: Fields with camundaVar=true (e.g. action, severity, riskScore) are orchestration variables — they drive FEEL gateway decisions. Fields with camundaVar=false (e.g. analystName, iocList, affectedSystems) are domain object data — written to WLC PostgreSQL only and never cross into Camunda.
